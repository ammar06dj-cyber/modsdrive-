# دليل إعداد وتشغيل البيئة الآمنة (Secure Environment Setup)

يوضح هذا الدليل الخطوات والتدابير الأمنية المتبعة لتشغيل وتأمين خادم التطبيق، وضمان حماية لوحة التحكم والبيانات الحساسة من الوصول غير المصرح به.

---

## 1. إعداد المتغيرات والبيئة الحساسة باستخدام `.env`

تمت تهيئة إعدادات التجاهل في نظام Git لمنع رفع ملفات البيئة وحظر تسريب بيانات الاعتماد الحساسة. يتم تقديم ملف نموذج باسم `.env.example` ليكون المرجع الأساسي لتكوين البيئة.

### خطوات إعداد ملف البيئة المحلية:
1. قم بإنشاء ملف جديد في المجلد الرئيسي للمشروع باسم `.env`.
2. انسخ محتويات الملف المرجعي `.env.example` وعيّن كلمة مرور الإدارة القوية الخاصة بك:

```env
# كلمة المرور الخاصة بلوحة التحكم الإدارية
ADMIN_PASSWORD=your_super_secure_and_long_password_here
```

### معايير وقواعد كلمة المرور المطلوبة:
- **الطول الأدنى**: يجب ألا يقل طول كلمة المرور عن 8 أحرف.
- **الطول الأقصى**: يجب ألا يتجاوز طول كلمة المرور 128 حرفاً لمنع هجمات الاستهلاك المفرط للذاكرة وموارد المعالجة.
- **الرموز المدعومة**: تدعم الحروف الإنجليزية الكبيرة والصغيرة، الأرقام، والرموز الخاصة الشائعة.
-  **ملاحظة هامة جداً**: تم إلغاء كافة كلمات المرور الافتراضية للتطوير تماماً. في حال عدم تكوين المتغير `ADMIN_PASSWORD` في السيرفر، ستفشل جميع الطلبات الموجهة للوحة الإدارة وسيعطي الخادم خطأ `500 Server misconfiguration` مع إظهار رسالة تنبيه واضحة في سجلات الخادم (Console).

## 2. متطلبات التشفير وفرض اتصال HTTPS الآمن

لحماية قنوات الاتصال والبيانات العابرة للشبكة، تم دمج آلية توجيه آمنة تفرض التشفير الكامل:

- عند تشغيل التطبيق في بيئة الإنتاج (`NODE_ENV === "production"`)، يقوم خادم التطبيق تلقائياً بالتحقق من بروتوكول الاتصال الوارد.
- يتم ترحيل وإعادة توجيه كافة طلبات الاتصال غير المشفرة (HTTP) بشكل فوري وسلس عبر رمز الحالة `301` إلى الرابط المماثل الآمن والمشفّر عبر بروتوكول **HTTPS**.
- نوصي باستخدام شهادات Let's Encrypt أو استخدام خدمات البوابات الآمنة التي يوفرها موفرو الخدمة السحابية (مثل Cloud Run / Cloudflare Load Balancer) لإدارة وتجديد الشهادات الموثوقة بسهولة.

## 3. محدد المعدل لحماية الخادم ومقاومة الهجمات (Rate Limiter)

يستخدم الخادم محدد المعدل لحظر الهجمات المتكررة ومنع تجاوز سعة معالجة موارد الخادم.

### جدول معايير وإعدادات محدد المعدل (Rate Limiter):

| نوع المسار المستهدف | مسار نقطة النهاية (Endpoint Route) | الحد الأقصى للطلبات (Max Requests) | الفترة الزمنية للمراقبة (Time Window) | السلوك والرد عند تجاوز الحد الأقصى |
| :--- | :--- | :--- | :--- | :--- |
| **مصادقة لوحة الإدارة** | `/api/admin-auth` | **5 طلبات كحد أقصى** | **15 دقيقة** | حظر فوري وإرجاع رمز الحالة `429 Too Many Requests` |
| **المسارات والطلبات العامة الأخرى** | `*` (جميع المسارات المتبقية) | **100 طلب كحد أقصى** | **15 دقيقة** | حظر مؤقت وإرجاع رمز الحالة `429 Too Many Requests` |

---

### كيفية تعديل وتخصيص إعدادات محدد المعدل:

يمكنك بسهولة تعديل وتخصيص هذه المعايير داخل ملف `server.ts` عبر تحرير إعدادات محددات المعدل المطابقة للمتطلبات كالتالي:

1. **تحديث محدد مصادقة الإدارة (Admin Auth Limiter)**:
   ```typescript
   const adminAuthLimiter = rateLimit({
     windowMs: 15 * 60 * 1000, // المدة الزمنية بالملي ثانية (15 دقيقة حالياً)
     max: 5, // عدد المحاولات القصوى الممنوحة لكل عنوان IP
     message: { error: "لقد تجاوزت الحد الأقصى للمحاولات المسموح بها، يرجى المحاولة بعد 15 دقيقة." },
     standardHeaders: true,
     legacyHeaders: false,
   });
   ```

2. **تحديث محدد الطلبات العام (General Limiter)**:
   ```typescript
   const generalLimiter = rateLimit({
     windowMs: 15 * 60 * 1000, // 15 دقيقة بالملي ثانية
     max: 100, // عدد الطلبات الأقصى المسموح به لكل عنوان IP
     message: { error: "طلبات كثيرة جداً من هذا العنوان، يرجى المحاولة مجدداً في وقت لاحق." }
   });
   ```

## 4. تعليمات تهيئة وإعداد Google Search Console للسitemap

لضمان أرشفة الموقع وسحب ملف الفهرسة الـ `sitemap.xml` بشكل صحيح ومقاومة خطأ "تعذر جلب الملف" (Couldn't Fetch)، يرجى تطبيق التعليمات التالية:

1. **إلحاق محرك Sitemap بشكل صحيح**:
   - داخل لوحة تحكم **Google Search Console**، عند إرسال ملف Sitemap، اكتب الاسم بالطريقة التالية تماماً:
     `sitemap.xml`
   - **لا تقم وبشدة** بكتابة slash مسبق (أي لا تكتب `/sitemap.xml`).

2. **التحقق من إعداد نطاق الخاصية (Property Domain)**:
   - تأكد أن نطاق الخاصية النشطة المحددة داخل Google Search Console يطابق تماماً رابط الدومين النهائي الآمن:
     `https://modsdrive.pages.dev`

3. **فحص قابلية الوصول المباشر**:
   - افتح الروابط التالية يدوياً من المتصفح والتحقق من فتحها مباشرة بدون أي عمليات تحويل أو إعادة توجيه (Redirects):
     - ملف الفهرسة: [https://modsdrive.pages.dev/sitemap.xml](https://modsdrive.pages.dev/sitemap.xml)
     - ملف التحكم بالزواحف: [https://modsdrive.pages.dev/robots.txt](https://modsdrive.pages.dev/robots.txt)

4. **في حال استمرار ظهور تنبيه "Couldn't Fetch" من قوقل**:
   - لا تتسرع، ففي كثير من الأحيان يعود السبب لتأخير برمجيات الزحف الخاصة بـ Googlebot. يرجى الانتظار لمدة 24 ساعة.
   - قم بحذف الـ Sitemap الحالية من اللوحة، ثم أعد إرسالها باسم `sitemap.xml` (بدون أي بادئات).
   - استخدم أداة **URL Inspection** لفحص الرابط بشكل مباشر واختبار ما إذا كان الرابط متاحاً للزحف الفوري.
   - تأكد من عدم وجود خيارات تقييد صارمة أو حظر من بوتات قوقل في جدار الحماية الخاص بـ Cloudflare (مثل Bot Fight Mode / Web Application Firewall WAF).
